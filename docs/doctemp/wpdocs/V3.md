## Important Research Constraint
Based on the current OpenAI platform guidance, the long-term roadmap should stay aligned with what OpenAI officially supports today:
- optimization should begin with **evals first**
- prompt improvements should happen before training
- supervised fine-tuning currently supports supported OpenAI base-model families rather than arbitrary frontier snapshots
- distillation is an official path for transferring stronger-model behavior to smaller models
- reinforcement fine-tuning is intended for harder reasoning tasks and depends on grader quality

That means V3 should be architected as an **OpenAI-powered proprietary model program**, not as marketing language that implies unsupported training claims.

## Official OpenAI Optimization Flow Applied To LyraAlpha
The research-backed V3 sequence should follow the OpenAI optimization flywheel.

### Phase 1 — Evals-first foundation
Before training anything:
- define the critical tasks
- create holdout datasets
- build graders
- establish baseline scores for current prompts and models

Key principle from OpenAI guidance:
- do **evals first** so every later tuning decision has a measurable target

**Current state (March 2026):** A live prompt pipeline audit harness is already in place.
- `scripts/audit-prompt-pipeline.ts` runs 27 real end-to-end test cases across all plan/tier combinations against the live Azure OpenAI stack, measuring quality score (0–100), TTFT, e2e latency, word-budget compliance, cost, and truncation detection
- `scripts/compare-audit-runs.ts` diffs two audit runs (baseline vs current) and generates a regression/improvement report with MAJOR/minor severity flagging
- March 2026 baseline: 27/27 pass, avg quality 97.8/100, 0 truncations, hybrid routing 36% cheaper than all-full

This is the operational foundation for Phase 1. The next step is expanding from latency/quality/cost tracking into finance-domain graders (factual grounding, regime reasoning, schema adherence).

### Phase 2 — Prompt optimization and workflow cleanup
Improve performance without training first:
- tighten prompts
- improve retrieval quality
- strengthen output contracts
- add completeness and verification contracts
- improve structured outputs and tool discipline

This phase matters because OpenAI explicitly recommends improving prompts before investing heavily in training.

### Phase 3 — Supervised fine-tuning (SFT)
Use SFT for tasks where we can clearly show the model what “good” looks like.

Best V3 use cases:
- fixed answer formats
- support correctness
- routing consistency
- finance explanation style control
- schema-constrained outputs
- citation and disclosure formatting

OpenAI guidance indicates:
- start with evals already in place
- begin with a relatively small but high-quality dataset
- improvements often show up with roughly 50–100 strong examples
- if 50 strong examples do not help, revisit the task/prompt before simply adding more data

### Phase 4 — Distillation
Distill strong frontier-model behavior into smaller, cheaper custom models.

This should become one of the highest-leverage V3 tactics.

Recommended flow based on OpenAI guidance:
- get strong performance with the best teacher prompt and model
- store or capture high-quality outputs
- filter those outputs through eval thresholds
- use the passing outputs to train smaller student models
- run evals again to compare student vs teacher

Why this matters for LyraAlpha:
- lower latency
- lower serving cost
- more stable production behavior
- faster iteration on specialized workflows

### Phase 5 — Preference optimization / ranking alignment
For workflows where “better” is subjective rather than strictly deterministic, use preference-based fine-tuning methods.

Good candidates:
- tone selection
- explanation ranking
- premium answer quality preferences
- choosing the best summary among multiple valid options

### Phase 6 — Reinforcement fine-tuning (RFT)
Use RFT only for truly hard reasoning tasks where grading quality is strong enough to support it.

Potential long-term use cases:
- complex portfolio reasoning
- multi-step risk interpretation
- conflict resolution between competing market signals
- determining the most relevant evidence for a premium analytical answer

Important OpenAI-aligned cautions:
- grader quality is critical
- reward hacking is a real risk
- final behavior must be inspected on representative prompts before deployment
- OpenAI safety checks do not replace our own domain evaluations

This should be a later V3 phase, not the starting point.

## Fintech-Specific Data Program
A serious proprietary model program requires a disciplined data strategy.

### Data sources we should prioritize
- top-performing historical user prompts
- expert-annotated financial answers
- deterministic score + explanation pairs
- premium workflow examples
- onboarding/support transcripts
- failure cases from production logs
- negative examples showing what the model must not do
- finance-domain structured outputs for compare, screen, summarize, and explain tasks

### Data curation rules
- include realistic production-style prompts
- keep training and holdout sets separate
- preserve diversity across assets, regions, and user sophistication levels
- label edge cases deliberately
- include compliance-boundary prompts, not just successful happy-path examples
- do not train on third-party data without rights that allow the intended usage

## Proprietary Evaluation Program
The eval program should become a permanent product capability, not a one-off research task.

### V3 eval tracks
- **Analysis fidelity evals**
  - does the answer respect the deterministic score inputs?
- **Regime reasoning evals**
  - does the answer adapt correctly to market regime?
- **Portfolio logic evals**
  - does the answer reason coherently about diversification, overlap, and tradeoffs?
- **Support correctness evals**
  - does Myra return correct plan, credits, and policy guidance?
- **Compliance-boundary evals**
  - does the system avoid personalized financial-advice drift?
- **Format evals**
  - does the output meet schema/citation requirements?
- **Latency/cost evals**
  - does the optimized model improve economics without unacceptable quality loss?

### Release gate
No custom model should move into production unless it beats the baseline on:
- quality
- stability
- policy compliance
- cost or latency for its target workload

## Risks and Guardrails
### Risk 1 — overstating what is “ours”
Guardrail:
- say proprietary model layer or proprietary fintech intelligence system, not self-trained frontier foundation model

### Risk 2 — training before measurement
Guardrail:
- evals and graders come first

### Risk 3 — low-quality datasets
Guardrail:
- prioritize high-quality examples over raw volume

### Risk 4 — reward hacking and overfitting
Guardrail:
- maintain holdout sets, inspect representative prompts, and compare to baseline regularly

### Risk 5 — compliance drift
Guardrail:
- include finance-boundary and refusal behavior in every evaluation track

### Risk 6 — third-party data misuse
Guardrail:
- train only on data we own, generate, or are explicitly licensed to use for this purpose
